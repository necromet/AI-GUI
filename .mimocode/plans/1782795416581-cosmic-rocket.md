## 2. Gaps & Issues Found

### GAP 1: Dead Code â€” `services/mimoService.ts` (361 lines, unused)

**Severity**: Low (code smell)

No component imports from `mimoService.ts`. All imports point to `apiService.ts`. The legacy service has retry logic, quota detection, and direct-to-MiMo API calls that are completely unused. This is 361 lines of dead code.

**Includes features not in apiService.ts**:
- Retry with exponential backoff on 429
- Quota exhaustion detection
- `abortableDelay` helper

### GAP 2: Dead Code â€” Stub Routes (`models.ts`, `conversations.ts`)

**Severity**: Low (code smell)

`GET/POST /api/models` and `GET/POST /api/conversations` are never called from any frontend component. Models and conversations are managed entirely client-side via IndexedDB (`databaseAdapter.ts` â†’ `databaseService.ts`). These 4 routes + 2 route files are dead code.

### GAP 3: No Frontend Health Check

**Severity**: Medium

`GET /api/health` exists server-side but no frontend code ever calls it. There's no way for the UI to detect whether the Express backend is running. If the backend is down, users get silent failures on chat/TTS/ASR with no diagnostic feedback.

### GAP 4: Duplicate Provider Config Logic

**Severity**: Low

`getProviderConfigFromBody()` in `server/routes/chat.ts:14-21` is a copy of `getProviderConfig()` in `server/services/mimoService.ts:17`. The transcribe route uses the inline copy; other routes use the service. Two identical implementations that could drift.

### GAP 5: No Production Proxy Config

**Severity**: High (for deployment)

The Vite dev server proxies `/api` â†’ `localhost:3001` and `/mimo-api`/`/mimo-direct-api` â†’ MiMo. In production (`npm run build`), the frontend is static files in `dist/` with no proxy. There's no nginx config, no Docker reverse proxy config, and no Express static file serving setup documented. The `/mimo-api` and `/mimo-direct-api` proxies become unreachable in production.

### GAP 6: No Express Global Error Handler

**Severity**: Medium

Each route has its own try/catch, but there's no `app.use((err, req, res, next) => ...)` middleware. Unhandled errors in middleware (e.g., multer, express.json) could produce generic Express error pages instead of JSON responses.

### GAP 7: CORS Wide Open

**Severity**: Medium (security)

`cors()` with zero options allows all origins, methods, and headers. In production, this should be restricted.

### GAP 8: No Auth on Express Routes

**Severity**: Medium (security)

API keys are server-side only (good â€” not exposed to client), but anyone who can reach port 3001 can use the chat/TTS/ASR endpoints. No authentication middleware exists.

### GAP 9: TTS Audio Object Memory Leak

**Severity**: Low

`generateSpeech()` in `apiService.ts:185` creates `URL.createObjectURL(blob)` but there's no corresponding `URL.revokeObjectURL()` anywhere in `TTSPanel.tsx`, `VoiceClonePanel.tsx`, or `VoiceDesignPanel.tsx`. Each TTS call leaks a blob URL.

### GAP 10: `mimoService.ts` Contains Features apiService.ts Lacks

**Severity**: Medium

The unused `mimoService.ts` has retry logic (429 backoff, 3 retries) and quota detection that `apiService.ts` does not. If the backend returns a 429, the frontend throws immediately with no retry. These features should either be ported to the server-side or added to `apiService.ts`.
